# Design: CI/Tests/Contributing Guide + Dark/Light Mode

Date: 2026-04-23

## Overview

Two independent features:
1. **CI pipeline + test suite + CONTRIBUTING.md** — protect contributors from breaking game logic; establish clear onboarding and PR workflow; integrate with Vercel's preview deployment model.
2. **Dark/light mode toggle** — add a sun/moon toggle to the header; centralize all color tokens; support both themes via CSS variables.

---

## Part 1: CI / Tests / CONTRIBUTING.md

### Branch Strategy

| Branch | Purpose | Vercel URL |
|---|---|---|
| `main` | Production | `dbt-quest.vercel.app` (or custom domain) |
| `preview` | Staging — persistent preview | `dbt-quest-git-preview-<team>.vercel.app` |
| `feature/*` or any PR branch | Per-PR ephemeral preview | auto-generated by Vercel |

Contributors branch off `main`, open PRs targeting `main`. The `preview` branch is used to accumulate changes for manual QA before merging to main.

### GitHub Actions — `.github/workflows/ci.yml`

Triggers: push to any branch, pull_request targeting `main`.

Steps (sequential):
1. `npm ci` — reproducible install
2. `npm run lint` — ESLint
3. `npm run build` — `tsc -b && vite build` (type errors fail the build)
4. `npm run test` — Vitest unit tests

All steps must pass for the check to go green. Vercel's own preview deployment is handled separately by its GitHub integration (no manual deploy step needed in CI).

### Test Framework — Vitest

Add `vitest` as a dev dependency. Add script `"test": "vitest run"` to `package.json`.

Test files live in `src/engine/__tests__/`:

- **`commandParser.test.ts`** — covers valid commands (`dbt run`, `dbt test`, `dbt run --select model+`), invalid/empty input, selector parsing edge cases.
- **`dagBuilder.test.ts`** — given a map of SQL files with `ref()` calls, asserts correct nodes and edges are produced.
- **`validators.test.ts`** — for each level validator, tests at least one passing state and one failing state to prevent regressions when engine internals change.

No E2E tests in scope for now. Vitest is chosen because it reuses the Vite config with zero additional setup.

### CONTRIBUTING.md (root of repo)

Sections:
1. **Prerequisites** — Node 20+, npm
2. **Local setup** — `npm ci`, `npm run dev`
3. **Running tests** — `npm run test` (Vitest), `npm run build` (type check + build)
4. **Branch & PR workflow** — branch off `main`, open PR, CI runs automatically, Vercel posts a preview URL comment
5. **Adding a level** — explain the `Level` type, `initialFiles`, `goal`, `seeds`, `validate()`, point to an existing level as reference
6. **Code style** — TypeScript strict, no CSS-in-JS, Tailwind only, no comments unless the why is non-obvious

## Part 2: Dark/Light Mode

### Architecture

**Single source of truth: CSS variables in `src/index.css`**

The existing `@theme` block defines dark mode (default). A `[data-theme="light"]` override block defines the light palette. No JS theme object is needed — CSS variables are the contract between the theme system and components.

Dark palette (existing):
```
--color-base:         #0d1117
--color-surface:      #161b22
--color-border:       #30363d
--color-muted:        #484f58
--color-text:         #e6edf3
--color-text-muted:   #7d8590
--color-accent-orange: #ff694a
--color-accent-orange-dim: #ff694a33
--color-success:      #3fb950
--color-fail:         #f85149
```

Light palette (new):
```
--color-base:         #ffffff
--color-surface:      #f6f8fa
--color-border:       #d0d7de
--color-muted:        #8c959f
--color-text:         #1f2328
--color-text-muted:   #636c76
--color-accent-orange: #e05a3a   (slightly deeper for contrast on white)
--color-accent-orange-dim: #e05a3a22
--color-success:      #1a7f37
--color-fail:         #cf222e
```

### Component Migration

All hardcoded hex strings in inline styles across components are replaced with `var(--color-*)` references. Affected files: `Header.tsx`, `Editor.tsx`, `DagPanel.tsx`, `TerminalPanel.tsx`, `BottomPanel.tsx`, `LevelIntroModal.tsx`, `LevelPanel.tsx`, `ResultsPanel.tsx`, `FileExplorer.tsx`, `DatabaseExplorer.tsx`.

Monaco Editor: pass `theme={isDark ? 'vs-dark' : 'light'}` to `<Editor>`.

xterm.js: update the `theme` option passed to the `Terminal` constructor when the theme changes. Since xterm doesn't observe CSS variables, theme is passed explicitly from the store.

### Theme State

Add `theme: 'dark' | 'light'` and `toggleTheme: () => void` to `gameStore`. On toggle:
1. Flip `document.documentElement.dataset.theme` (`''` for dark, `'light'` for light)
2. Persist to `localStorage` under key `dbt-quest-theme`

On app init (`main.tsx`): read `localStorage`, fall back to `window.matchMedia('(prefers-color-scheme: light)')`, set `document.documentElement.dataset.theme` before first render.

### Toggle Button

Sun/moon icon button added to `Header`, placed between the progress bar and the help button. Uses existing `ActionButton`-style inline styling. Icon swaps based on current theme.

### Scope

Light mode will be fully functional and readable. It is not a pixel-perfect redesign — color choices are pragmatic and dbt-branded, not validated by a designer.

---

## Out of Scope

- Mobile layout (already out of scope per CLAUDE.md)
- E2E / Playwright tests
- A separate `theme.ts` JS file (CSS variables are sufficient as the contract)
- Automated Lighthouse / accessibility CI checks
